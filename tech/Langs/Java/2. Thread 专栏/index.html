    2. Thread 专栏

    状态：blocked 与 waiting

    • WAIT - I'm waiting to be given some work, so I'm idle right now:  thread is waiting for a signal from another thread. This happens typically by calling Object.wait(), or Thread.join(). The thread will then remain in this state until another thread calls Object.notify(), or dies.
    • BLOCKED - I'm busy trying to get work done but another thread is standing in my way, so I'm idle right now: a thread is about to enter a synchronized block, but there is another thread currently running inside a synchronized block on the same object. The first thread must then wait for the second thread to exit its block.
     public static void main(String[] args) {

        // Runnable runnable = new Runnable() {
        //  @Override
        //  public void run() {
        //   longTask();
        //  }
        // };
        Thread thread2 = new Thread(() -> longTask());
     thread2.getState();
        thread2.start();
        System.out.println("Doing other work on main thread");

    }

    interrupt a thread

    • interrupt 方法(但是只是向 t 线程发出了请求，t 线程是否响应，需要看具体代码)
    • 由此可以知道，线程其实没有父子关系，它们是平级的
    • 如果有一个线程没有退出，JVM进程就不会退出
    • 也可以设置线程的标志位，但是需要 volatile 关键字标记。在 java 的内存模型中，（变量的值保存在主内存中），每个线程有自己的内存区，会缓存进程内对应内存的值（当线程访问变量时，会获取一个副本，并保存在自己的工作内存中），并不会实时更新（如果线程修改了变量的值，虚拟机会在某个时刻把修改后的值回写到主内存，但是，这个时间是不确定的）。

    守护进程

    t.setDaemon(true); 不影响 ~~主进程~~(JVM 进程 / 虚拟机) 退出

    线程同步

    Cause：a += 1 这个操作实际包含了三步（Load Add Store），亦既这并不是一个原子操作

    synchronized

    解决： - synchronized keyword: 它以 object（相当于钥匙） 为入参，if when you put synchronized on the method, it basically means the same as synchronized (this), that is, a synchronization on the object this method is executed on (也就等同于在 method 内部调用 synchronized(this){}). concurrency - java - what does synchronized really do according the java memory model? - Stack Overflow Chapter 17. Threads and Locks

    synchronized vs. volatile
    • volatile 保证时效性不是原子性（如果本地线程的变量更改，可以第一时间同步到主内存；但之前已经获取过主内存变量的本地副本的线程，无法得知主内存中这个变量有更改）
    • synchronized 某种程度上说包括了 volatile 的功能。synchronized除了加锁外，还具有内存屏障功能，并且强制读取所有共享变量的主内存最新值，退出synchronized时再强制回写主内存（if modified）
    线程安全的类

    即类自己对变量做了加锁处理，可被多线程~~安全~~正确访问 thread-safe. StringBuffer 就是线程安全的

    wait 和 notify

    一个好例子：

    import java.util.*;

public class Main {
    public static void main(String[] args) throws InterruptedException {
        var q = new TaskQueue();
        var ts = new ArrayList<Thread>();
        for (int i=0; i<5; i++) {
            var t = new Thread() {
                public void run() {
                    // 执行task:
                    while (true) {
                        try {
                            String s = q.getTask();
                            System.out.println("execute task: " + s);
                        } catch (InterruptedException e) {
                            return;
                        }
                    }
                }
            };
            t.start();
            ts.add(t);
        }
        var add = new Thread(() -> {
            for (int i=0; i<10; i++) {
                // 放入task:
                String s = "t-" + Math.random();
                System.out.println("add task: " + s);
                q.addTask(s);
                try { Thread.sleep(100); } catch(InterruptedException e) {}
            }
        });
        add.start();
        add.join();
        Thread.sleep(100);
        for (var t : ts) {
            t.interrupt();
        }
    }
}

class TaskQueue {
    Queue<String> queue = new LinkedList<>();

    public synchronized void addTask(String s) {
        this.queue.add(s);
        this.notifyAll(); // 唤醒其他等待的消费者
    }

    public synchronized String getTask() throws InterruptedException {
        // 注意这里的 while, 因为被唤醒后，还需要判断 queue 里是否有资源可消费（也许已经被别人消费完了）
        while (queue.isEmpty()) { 
            this.wait();
        }
        return queue.remove();
    }
}

    锁

    可重入锁(通俗意义的)

    同一个线程重复获取同一个锁。这样不但要判断是否是第一次获取，还要记录这是第几次获取。每获取一次锁，记录+1，每退出synchronized块，记录-1，减到0的时候，才会真正释放锁。

    ReentrantLock

    Java 5 引入，可以设置重试次数

    if (lock.tryLock(1, TimeUnit.SECONDS)) {
    try {
        ...
    } finally {
        lock.unlock();
    }
}

    hh 我之前写的抽奖的乐观锁也是这样，可以重试！ Lock.Condition提供的 await()、signal()、signalAll() 原理和synchronized锁对象的wait()、notify()、notifyAll()是一致的，并且其行为也是一样的。

    ReadWriteLock
    • 允许多个线程同时读
    • 如果有一个线程在写（入），则其他线程都需要等待该线程写完后才能读（取）某种意义上的悲观读锁
    public class Counter {
    private final Lock lock = new ReentrantLock();
    private int[] counts = new int[10];

    public void inc(int index) {
        lock.lock();
        try {
            counts[index] += 1;
        } finally {
            lock.unlock();
        }
    }

    public int[] get() {
        lock.lock();
        try {
            return Arrays.copyOf(counts, counts.length);
        } finally {
            lock.unlock();
        }
    }
}

    有一个问题很好，既然写锁住了，那为什么还需要读锁？ 因为读锁的目的不是因为读的数据是错的，而是为了保证连续读逻辑上一致：

    int x = obj.x;
// 这里线程可能中断
int y = obj.y;

    同理，如果写入是原子操作，如赋值，那么加锁的目的是为了保证逻辑正确（比如连续写入两个）。

    StampedLock

    目的：相比 ReadWriteLock, 提高读取效率，Java 8引入。

    乐观读锁。在读取前会先通过 stampedLock.tryOptimisticLock() 获取一个版本号，在关键操作前，通过 stampedLock.tryOptimisticLock(stamp) 校验最新版本号是否发生变化，然后再通过 stampedLock.readLock() 获取一个悲观读锁。

    Semaphore

    限制数量的锁。应用场景：保证同一时刻最多有 n 个线程可以访问的受限资源。

    Concurrent 集合

    应用场景：BlockingQueue. 何为 blockQueue? 当一个线程从 queue 里取任务时(queue.getTask())，如果 queue 没有内容，线程会变为等待状态，直到有内容后获取并返回。

    如果通过 ReentrantLock 和 Condition 来实现：

    public class TaskQueue {
    private final Lock lock = new ReentrantLock();
    private final Condition condition = lock.newCondition();
    private Queue<String> queue = new LinkedList<>();

    public void addTask(String s) {
        lock.lock();
        try {
            queue.add(s);
            condition.signalAll();
        } finally {
            lock.unlock();
        }
    }

    public String getTask() {
        lock.lock();
        try {
            while (queue.isEmpty()) {
                condition.await();
            }
            return queue.remove();
        } finally {
            lock.unlock();
        }
    }
}
    Concurrent.Atomic

    原子操作的封装类。无锁(lock-free)，利用了 CAS，有点乐观锁的意味。

    自己模拟 incrementAndGet()：

    public int incrementAndGet(AtomicInteger var) {
    int prev, next;
    do {
        prev = var.get();
        next = prev + 1;
    } while ( ! var.compareAndSet(prev, next));
    return next;
}

    应用场景：多线程安全的递增 ID 生成器

    class IdGenerator {
    AtomicLong var = new AtomicLong(0);

    public long getNextId() {
        return var.incrementAndGet();
    }
}
